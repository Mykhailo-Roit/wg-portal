# Анализ функционала продукта `wg-portal`

## Контекст анализа

Этот документ собран по результатам анализа репозитория в ветке `feature/SCIM_deprovisioning`.
Описание основано на фактической реализации продукта в:

- backend на Go (`internal/...`, `cmd/wg-portal/...`)
- frontend на Vue (`frontend/src/...`)
- встроенной документации (`README.md`, `docs/...`)

Продукт представляет собой **WireGuard Portal**: self-hosted веб-портал для управления WireGuard-инфраструктурой, пользователями, peer-конфигурациями и связанными процессами доступа.

## Что это за продукт

Продукт решает задачу централизованного управления WireGuard через:

- веб-интерфейс для администраторов и обычных пользователей
- REST API для автоматизации и внешних клиентов
- механизмы self-service provisioning
- интеграцию с LDAP / OAuth2 / OIDC / Passkeys
- экспорт конфигураций, QR-кодов и отправку по email
- аудит, статистику и Prometheus-метрики
- webhook-интеграции
- поддержку нескольких backend’ов WireGuard

По сути, это административная и пользовательская панель поверх WireGuard, которая управляет интерфейсами, peer’ами, учетными записями и доступом.

## Основные сущности домена

### 1. Интерфейс WireGuard

Система умеет хранить и управлять несколькими WireGuard-интерфейсами.

Поддерживаются режимы:

- `server`
- `client`
- `unknown` для импортированных/неопределённых интерфейсов

Для интерфейса доступны:

- идентификатор и отображаемое имя
- ключи
- адреса IPv4/IPv6
- порт прослушивания
- DNS и search domains
- MTU
- firewall mark
- routing table
- pre/post up/down hooks
- включение/отключение
- backend, на котором живёт интерфейс
- набор дефолтных параметров для создаваемых peer’ов
- флаг сохранения конфигурации в файл
- флаг автосоздания default peer для новых пользователей

### 2. Peer

Peer является клиентской или серверной точкой WireGuard, привязанной к интерфейсу и, как правило, к пользователю.

Для peer’а поддерживаются:

- публичный и приватный ключ
- preshared key
- display name
- связанный пользователь
- интерфейс
- endpoint
- allowed IPs и extra allowed IPs
- адреса интерфейса peer’а
- DNS / DNS search
- MTU
- keepalive
- firewall mark
- routing table
- pre/post up/down hooks
- статус отключения
- срок действия `ExpiresAt`
- отметка автоматического создания
- заметки

### 3. Пользователь

Система управляет локальными и внешними пользователями.

Поддерживаемые поля:

- уникальный identifier
- email
- имя и фамилия
- телефон
- department
- notes
- admin-флаг
- auth sources
- API token
- блокировка
- отключение
- WebAuthn credentials
- флаг сохранения локальных изменений при внешней синхронизации
- `externalId` для SCIM

### 4. Аудит и статистика

Есть отдельные модели для:

- audit entries
- статистики peer’ов
- статистики интерфейсов
- событий подключения/отключения

## Роли и модель доступа

Система явно разделяет роли:

- `Admin`
- `User`

### Возможности администратора

Администратор может:

- управлять всеми пользователями
- управлять всеми peer’ами
- создавать и редактировать интерфейсы
- использовать bulk-операции
- видеть audit log
- работать с административными API
- получать полный список интерфейсов, пользователей и статистики

### Возможности обычного пользователя

Обычный пользователь может:

- видеть свои peer’ы
- скачивать свои конфиги
- получать свои QR-коды
- использовать self-provisioning, если он включён
- менять пароль для локального аккаунта
- управлять своими passkeys
- включать/выключать собственный API token, если это разрешено настройками

### Ограничения self-provisioning

Из бизнес-логики видно, что self-provisioning:

- глобально включается настройкой `self_provisioning_allowed`
- для обычного пользователя разрешён только на интерфейсах типа `server`
- может быть дополнительно ограничен по списку разрешённых интерфейсов через LDAP materialization
- может ограничиваться по количеству дополнительных peer’ов на пользователя

## Пользовательский веб-интерфейс

Frontend реализован на Vue и даёт следующие разделы.

### 1. Home

Домашняя страница:

- показывает entry point для пользователя
- даёт быстрый переход в профиль
- даёт быстрый переход в административные разделы для admin
- содержит ссылки на документацию и установку WireGuard

### 2. Login

Экран входа поддерживает:

- логин по username/password
- вход через WebAuthn / Passkeys
- вход через внешние OAuth/OIDC провайдеры
- режим, когда локальная форма логина скрыта и остаются только внешние способы входа

### 3. Interfaces

Основной административный экран управления WireGuard.

Поддерживает:

- список интерфейсов
- выбор активного интерфейса
- создание нового интерфейса
- просмотр параметров интерфейса
- редактирование интерфейса
- удаление интерфейса
- просмотр backend’а интерфейса
- скачивание конфигурации интерфейса
- сохранение конфигурации интерфейса в `wg-quick` файл, если это поддерживается настройками
- просмотр текущего трафика интерфейса
- список peer’ов выбранного интерфейса

Для peer’ов внутри интерфейса доступны:

- просмотр карточки peer’а
- создание peer’а
- массовое создание peer’ов
- редактирование peer’а
- удаление peer’а
- bulk enable / disable / delete
- выбор нескольких peer’ов
- отображение состояния peer’а, срока действия и статистики

### 4. Users

Административный раздел управления пользователями.

Поддерживает:

- поиск
- пагинацию
- просмотр списка всех пользователей
- создание пользователя
- просмотр пользователя
- редактирование пользователя
- bulk delete
- bulk enable
- bulk disable
- bulk lock
- bulk unlock
- отображение источников аутентификации
- отображение количества peer’ов на пользователя
- отображение admin-статуса

### 5. Profile

Пользовательский раздел “мой профиль / мои устройства”.

Поддерживает:

- загрузку данных текущего пользователя
- просмотр списка своих peer’ов
- поиск и пагинацию
- просмотр статистики подключения по своим peer’ам
- просмотр RX/TX по peer’ам
- просмотр принадлежности peer’а к интерфейсу
- удаление своих peer’ов
- создание нового peer’а через self-provisioning
- редактирование своего peer’а через self-provisioning

### 6. Settings

Раздел пользовательских настроек включает:

- смену пароля для локальных DB-пользователей
- отображение минимальной длины пароля
- регистрацию новых passkeys
- просмотр списка passkeys
- переименование passkeys
- удаление passkeys
- включение API token
- отключение API token
- просмотр идентификатора пользователя и API token
- ссылку на документацию API

Поведение API token зависит от настройки `api_admin_only`.

### 7. Audit

Отдельный экран аудита для администратора.

Поддерживает:

- просмотр записей аудита
- фильтрацию поиском
- пагинацию
- отображение времени, severity, пользователя, origin и сообщения

### 8. Key Generator

Встроенный утилитарный экран.

Позволяет локально в браузере:

- генерировать X25519 keypair
- генерировать preshared key

Ключи генерируются на стороне клиента и не отправляются на сервер.

### 9. IP Calculator

Встроенный калькулятор allowed IPs.

Позволяет:

- задать список разрешённых сетей
- задать список исключаемых сетей
- автоматически вычесть исключаемые CIDR из allowed CIDR
- быстро исключить private ranges

Это вспомогательный инструмент для подготовки конфигураций.

## Управление интерфейсами и peer’ами

Серверная логика показывает, что продукт умеет не только CRUD-операции, но и прикладные сценарии управления WireGuard.

### Функции по интерфейсам

- подготовка нового интерфейса с дефолтными параметрами
- создание интерфейса
- обновление интерфейса
- удаление интерфейса
- получение интерфейса вместе с peer’ами
- получение полного списка интерфейсов
- генерация текстовой конфигурации интерфейса
- сохранение конфигурации интерфейса на диск
- применение обновлённых peer defaults ко всем peer’ам интерфейса

### Функции по peer’ам

- подготовка нового peer’а с автоматической генерацией ключей
- автоматический выбор свободного IP из пула интерфейса
- создание peer’а
- массовое создание peer’ов
- обновление peer’а
- удаление peer’а
- включение/отключение peer’а
- bulk delete / enable / disable
- получение текстового конфига peer’а
- получение QR-кода peer’а
- отправка peer-конфига по email

### Автоматизация создания peer’ов

Есть несколько удобных механизмов:

- автогенерация ключей
- автогенерация IP
- автоподстановка дефолтных настроек из интерфейса
- создание default peer для новых пользователей на серверных интерфейсах, где это включено
- массовое создание peer’ов под список пользователей

### Поддержка срока действия

Peer может иметь `ExpiresAt`, то есть продукт поддерживает сценарий временных доступов.

## WireGuard backends

Продукт абстрагирует backend WireGuard и умеет работать не только локально.

### Поддерживаемые backend’ы

- `local` через Linux WireGuard / wgctrl
- `MikroTik` через RouterOS REST API
- `pfSense` через pfSense REST API

### Что это даёт

- один портал может управлять несколькими backend-источниками
- интерфейс хранит привязку к backend’у
- новые интерфейсы создаются на default backend либо на выбранном пользователем backend’е
- UI показывает доступные backend’ы и валидирует существование backend’а у интерфейса

### Ограничения backend’ов

Из документации и кода видно:

- `MikroTik` помечен как beta
- `pfSense` помечен как alpha
- для `pfSense` часть статистики недоступна через API
- не все advanced-возможности одинаково доступны на каждом backend’е

## Аутентификация и identity management

Продукт имеет развитый identity layer.

### Поддерживаемые способы входа

- локальные учётные записи в БД
- LDAP
- OAuth2
- OIDC
- WebAuthn / Passkeys

### Возможности слоя аутентификации

- несколько внешних провайдеров одновременно
- auto-registration пользователей при первом успешном входе
- скрытие локальной формы логина
- маппинг admin-ролей по внешним атрибутам и группам
- ограничение входа по allowed domains для OIDC/OAuth
- ограничение входа только существующими пользователями
- единый локальный профиль пользователя при логине через разные источники

### LDAP-возможности

LDAP-слой поддерживает не только login, но и полноценную синхронизацию:

- периодический LDAP sync
- auto-create пользователей из LDAP
- disable missing users
- auto re-enable пользователей, если они снова появились
- materialization разрешённых интерфейсов через `interface_filter`
- возможность не перетирать локальные изменения пользователя

### WebAuthn / Passkeys

Поддерживаются:

- discoverable login
- регистрация нескольких passkeys на пользователя
- переименование credential’ов
- удаление credential’ов
- запрет операций с passkeys для заблокированных или отключённых пользователей

## API и программный доступ

Продукт предоставляет как минимум два REST API слоя плюс SCIM.

### API v0

Это внутренний / UI-oriented API, который обслуживает frontend.

Поддерживает:

- configuration / frontend settings
- web session и web authentication
- users CRUD
- interfaces CRUD
- peers CRUD
- bulk-операции
- audit entries
- config download
- config mail
- websocket endpoint

Этот слой тесно связан с веб-приложением.

### API v1

Это более внешний API для интеграций и автоматизации.

Поддерживает:

- user endpoints
- interface endpoints
- peer endpoints
- provisioning endpoints
- metrics endpoints

Есть встроенная OpenAPI/Swagger-документация через `doc.html`.

### Provisioning API

Отдельно выделен сценарий provisioning:

- получить информацию о пользователе и его peer’ах
- получить конфиг peer’а
- получить QR-код peer’а
- создать новый peer программно

При этом соблюдается модель прав:

- обычный пользователь работает только со своими объектами
- админ может работать со всеми
- self-provisioning влияет на доступность создания peer’ов

### API tokens

Пользователь может включать/выключать персональный API token.

Есть режим `api_admin_only`, при котором полноценный API-доступ разрешён только администраторам.

## SCIM v2.0

В текущей ветке реализован отдельный SCIM endpoint.

### Что поддерживается

Маршруты монтируются под `/scim/` и построены через библиотеку SCIM server.

Реализована работа с ресурсом `Users`:

- `Create`
- `Get`
- `GetAll`
- `Replace`
- `Patch`
- `Delete`

### Поддерживаемые SCIM-атрибуты

- `userName`
- `externalId`
- `active`
- `displayName`
- `name.givenName`
- `name.familyName`
- `emails`
- `phoneNumbers`

### Аутентификация SCIM

- bearer token
- constant-time compare токена
- доступ выполняется в контексте system admin

### Важная особенность этой ветки

Удаление пользователя через SCIM конфигурируемое:

- `disable` по умолчанию: пользователь не удаляется физически, а деактивируется с причиной `SCIM deprovisioned`
- `delete`: пользователь удаляется физически

Это ключевое поведение ветки `feature/SCIM_deprovisioning`.

## Email и доставка конфигурации

Продукт умеет рассылать WireGuard-конфигурации по email.

Поддерживаются сценарии:

- письмо с attachment-конфигом
- вложенный QR-code
- link-only режим

Логика отправки может:

- брать email из профиля пользователя
- при определённых настройках использовать `UserIdentifier` как email, если он валиден

Это полезно для self-service onboarding.

## Webhooks

Система поддерживает webhook-уведомления во внешние сервисы.

### Поддерживаемые сущности

- `user`
- `peer`
- `interface`
- `peer_metric`

### Поддерживаемые события

- `create`
- `update`
- `delete`
- `connect`
- `disconnect`

### Что это позволяет

- интеграцию с CMDB / IAM / automation pipelines
- реакцию на изменение пользователей и конфигураций
- реакцию на VPN connect/disconnect

Webhook отправляется как HTTP POST с JSON payload и опциональной авторизацией через header.

## Метрики и мониторинг

Продукт умеет собирать и публиковать статистику работы WireGuard.

### В UI и API видны

- connected / disconnected status
- last handshake
- bytes received / transmitted
- интерфейсные RX/TX показатели

### Prometheus

Есть отдельный metrics server, публикующий:

- `wireguard_interface_received_bytes_total`
- `wireguard_interface_sent_bytes_total`
- `wireguard_peer_last_handshake_seconds`
- `wireguard_peer_received_bytes_total`
- `wireguard_peer_sent_bytes_total`
- `wireguard_peer_up`

Также есть готовый Grafana dashboard в Helm chart assets.

## Аудит

Аудит в продукте является самостоятельной функциональной подсистемой.

Судя по структуре кода, аудит покрывает:

- действия пользователей
- действия системных сервисов
- security-sensitive операции
- административные изменения

В UI аудит доступен как отдельный экран, а в API как отдельный endpoint.

## Дополнительные системные возможности

### 1. Route management

Есть подсистема route manager, которая старается воспроизводить поведение `wg-quick` для default routes:

- установка маршрутов
- удаление маршрутов
- реакция на события через event bus

### 2. Event bus architecture

Подсистемы связаны через event bus:

- audit
- webhooks
- route manager
- статистика
- изменения user / peer / interface

Это означает, что продукт не просто CRUD-панель, а событийно-ориентированная управляющая система.

### 3. Мультиязычность

Frontend содержит переводы как минимум для:

- English
- German
- Russian
- Ukrainian
- French
- Spanish
- Portuguese
- Chinese
- Korean
- Vietnamese

То есть продукт ориентирован на международное использование.

## Замеченные ограничения и границы продукта

По README и коду видно, что продукт сознательно **не** делает некоторые вещи:

- не генерирует firewall rules автоматически
- не поддерживает произвольные ОС, основной сценарий завязан на Linux-hosted управление
- не импортирует приватные ключи существующей WireGuard-конфигурации автоматически

Также часть backend-функций зависит от возможностей конкретной платформы.

## Итоговая оценка продукта

По функционалу это не просто UI для редактирования `wg0.conf`, а полноценная платформа управления WireGuard-доступом со следующими уровнями возможностей:

1. инфраструктурное управление интерфейсами и peer’ами
2. identity and access management
3. self-service onboarding пользователей
4. интеграция с enterprise auth и lifecycle management
5. внешняя автоматизация через API, webhooks и SCIM
6. эксплуатационные функции: аудит, метрики, email, multi-backend support

Ветка `feature/SCIM_deprovisioning` дополнительно расширяет продукт в сторону enterprise IAM-сценариев, потому что добавляет полноценный SCIM user lifecycle с настраиваемым поведением deprovisioning.
